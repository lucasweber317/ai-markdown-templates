# Assumptions

**Instructions:** Document durable assumptions about your product, customers, or release process that should guide release notes writing.

---

## Product Assumptions

Example entries:

- Customers prefer concise, action-oriented release notes over verbose explanations
- Most customers deploy updates within 2 weeks of availability
- Administrators are the primary readers; they relay information to end users
- Customers are familiar with [specific technical concepts] and don't need basic definitions

---

## Audience Assumptions

Example entries:

- Readers have technical proficiency in [domain/technology]
- Customers expect [specific tone: professional, conversational, formal]
- Release notes are read in [context: email, in-app changelog, documentation portal]

---

## Process Assumptions

Example entries:

- Release notes are published before updates are deployed
- Marketing team reviews notes for T1/T2 releases before publication
- All features mentioned have corresponding documentation
- Breaking changes require prominent callouts and migration guidance
